# Codex workflow and Build Week evidence

## Primary build session

Keep the majority of core implementation in one primary Codex session and preserve its `/feedback` session ID for Devpost. Use that session for architecture, AST extraction, graph delta, GitHub integration, model schemas, tests, evaluation, remediation, and demo reliability—not merely polishing.

## Repository contract

`AGENTS.md` defines the standing constraints Codex must follow. Significant decisions belong in `docs/DECISIONS.md`, including whether the human or model made the call and what evidence changed it.

## Product remediation flow

The example `@hedge fix HEDGE-NNN` workflow:

1. Accepts only one strict command.
2. Confirms the commenter has repository write permission.
3. Rejects fork PRs.
4. Reads the latest machine-readable Hedge report generated by GitHub Actions.
5. Delimits the finding as untrusted evidence beneath a trusted repair contract.
6. Runs `openai/codex-action@v1` with workspace-write sandboxing and no push credentials.
7. Captures a binary patch and final Codex report.
8. Transfers the artifact to a separate job without the OpenAI credential.
9. Opens a draft PR against the original PR branch.
10. Requires separate verification before closure.

## Remaining live proof

Before recording, install the workflows in the demo repository and capture:

- Original Hedge security diff.
- Authorized fix command.
- Codex draft remediation PR.
- Witness success on the vulnerable revision.
- Witness blocked on the repaired revision.
- Legitimate behavior passing.
- Updated graph/control state.
- Benign PR silence.

Do not claim the live sequence until it has actually run.
